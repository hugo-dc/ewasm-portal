Ewasm 1.0

Ewasm 1.0 is an umbrella term for a collection of projects with the goal of bringing Wasm to the Eth1 mainnet. You may find the specs in the design repo.

  • Hera (C++) and Hera.rs (Rust): The ewasm 1.0 virtual machine which can be integrated in existing Ethereum client compatible with EVMC. Hera has support to different WebAssembly interpreters and compilers such as Wabt, Binaryen, and WAVM.
  • Ewasm Precompiles: Wasm implementation of the Eth1 precompiled contracts
  • Sentinel Contract: A system contract that validates and injects gas metering to contracts at deploy time. Written in Rust and targetting Wasm.
  • Ewasm Tests: A suite of state tests for Wasm bytecodes adhereing to the Ewasm interface, runnable by Ewasm-compatible clients.
  • Block Explorer (Etherchain Light: Fork of Etherchain Light for visualizing blocks in the Ewasm testnet

Ewasm API

The Ewasm API is an interface exposed to contracts written in Wasm to access Ethereum environment data. Clients integrating Ewasm must implement these functions to be compliant.

  • Rust Ewasm API: Rust library which defines an Ewasm API that allows accessing Ewasm functions from contracts written in Rust.
  • Assemblyscript Ewasm API: Ewasm API for AssemblyScript

EVM to Ewasm

Ewasm could be introduced to Eth1 either as a replacement to EVM or as a second VM. Either way, these tools help reduce the burden on clients by enabling them to run EVM contracts through their Wasm engine so they don’t have to maintain 2 VMs.

  • Runevm: A Wasm contract that embeds an EVM interpreter. It allows clients with only Wasm support to execute EVM bytecode.
  • Evm2wasm: Transcompiles EVM code to Ewasm
  • YEVM: “Why EVM?” is an EVM compiler infrastructure, targetting Ewasm by utilising Yul as an intermediate step.

Testnet

All these projects allowed to run an Ethereum 1.0 Testnet with Ewasm Support. This means you could create contracts in a language targetting EVM (i.e. Solidity), or you could write contracts using a language targetting WebAssembly (Rust, C, AssemblyScript, etc), and live in the same testnet. The testnet is not being actively maintained anymore. However you can play with a Ewasm 1.0 Testnet in a local environment using this repository.

Wasm utilities

Some utilities developed for internal usage specific to Ewasm were extracted and published to benefit the wider Wasm ecosystem:

  • Wasm Chisel: Some useful utilities to transform WebAssembly binaries, with feasures such as: remapimports, trimexports, snip, deployer, etc.
  • qimalloc: Quick Incremental (Wasteful) Memory Allocator. This memory allocator will not release any memory. Its main use case is in short-lived environment, such as WebAssembly binaries.
  • Wasm Profiler: Utilities for profiling WebAssembly binaries

Metering

EVM comes with gas metering out-of-the-box to put a cap on the resources a node is expected to spend on a transaction. Wasm has no such notion by default. Wasm-metering was the first attempt at introducing metering to Wasm. Since then various alternatives have been explored and compared, including:

  • Upperbound
  • Symbolic metering
  • Metering Injection
  • Atomic Metering
    • Bounded Loop Finder?
    • Taint Analysis
  • Wasm BluePrint
  • Metering by counting cycles in Boom
    • https://github.com/poemm/assembles

To learn more about the design considerations for metering see here.

  • EVMC: Ethereum Client-VM Connector API

Phase 2

Phase 2 brings execution to the data shards introduced in phase 1. We explored various designs for state execution on the shards, communication between the shards, etc. A notable project is Scout which facilitated prototyping and testing Wasm-based Execution Environments (EEs). Scout was first implemented in Rust and later ported to various other languages such as C++ and Typescript.

Execution Environments (EEs)

Scout enabled us and other researchers to rapidly prototype and assess the feasiblity of various EEs:

  • Stateless KMM Token: Stateless merkle token with optimal witnesses in Wasm
  • Groth16 verifier: Groth16 proof verifier written in Webassembly
  • Daiquiri: Eth2 Coin Mixer EE
  • Biturbo: Eth1 EE using Turboproofs
  • MiMC Merkle Tree: Binary merkle tree using MiMC hash function, implemented in Assemblyscript
  • MiMC Wasm: Implementation of ZKSNARK-friendly MiMC-Feistel hash function in Assemblyscript
  • turbo-mpas: Assemblyscript code to verify a Merkle branch
  • SMPT: Stateless MPT token EE using Parity Ethereum’s MPT Rust implementation
  • STARK Verifier: STARK verification of a number of MIMC rounds expressed inside an EE
  • SMT: Sparse merkle tree implementation in Typescript

Wasm Interpreters

  • Fizzy: Fizzy aims to be a fast, deterministic, and pedantic WebAssembly interpreter written in C++
  • PyWebAssembly: Implementation of the WebAssembly spec in Python
  • CppWebAssembly: C++ implementation of the WebAssembly spec
